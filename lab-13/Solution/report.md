## 2. Теоретическая база

### SAST-анализ безопасности
- **Статический анализ (SAST):** Сканирование исходных кодов программы без запуска ее выполнения. Помогает находить небезопасные паттерны программирования на самых ранних стадиях разработки.
- **Bandit:** Автоматический линтер кода Python, ищущий уязвимости по базе известных небезопасных сигнатур (вызовы shell, небезопасные SQL-запросы).
- **SQL-инъекция (SQLi):** Нарушение безопасности БД, когда ввод пользователя склеивается с телом SQL-запроса напрямую, позволяя злоумышленнику выполнять произвольные команды в СУБД. Решается параметризацией запросов.

### SCA-анализ зависимостей
- **Анализ состава ПО (SCA):** Мониторинг дерева импортируемых библиотек проекта для сопоставления их версий с базами данных раскрытых уязвимостей (CVE).
- **npm audit:** Инструмент командной строки для Node.js, анализирующий `package-lock.json` и строящий отчет об уровне угроз безопасности используемых пакетов.

## 4. Контрольные вопросы

1. **Различия SAST и DAST:**  
   SAST сканирует файлы исходного кода "изнутри" без запуска программы (White Box). DAST анализирует рабочее приложение "снаружи" в процессе выполнения (Black Box), имитируя атаки хакера.
2. **Как работает параметризованный запрос:**  
   БД компилирует структуру SQL-запроса заранее. Вводимые параметры передаются отдельно от команды, поэтому СУБД воспринимает их исключительно как строковые/числовые данные, а не как часть исполняемого SQL-кода.
3. **Что такое XSS и как защититься:**  
   Cross-Site Scripting — атака, при которой вредоносный JS-скрипт внедряется в HTML-страницу и выполняется в браузере других пользователей. Защита: экранирование спецсимволов при выводе (например, замена `<` на `&lt;`), валидация ввода и политика CSP.
4. **Зачем нужен SCA-анализ:**  
   Проекты часто импортируют сторонний код (библиотеки). SCA-анализ помогает отслеживать уязвимости в этих компонентах, так как уязвимость в импортируемом пакете подвергает риску все приложение.
