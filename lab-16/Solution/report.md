## Часть 1. Автоматизация разработки с помощью GigaChat

### Реализация API-клиента

Для работы с API GigaChat был подготовлен скрипт `gigachat_client.py`, который содержит:
- Класс настроек `GigaChatSettings`, инициализируемый из переменных окружения в `.env`;
- Класс `GigaChatSession`, отвечающий за авторизацию, получение JWT-токена доступа и отправку запросов к модели;
- Методы постобработки ответов и очистки markdown-разметки (`clean_markdown_code`).

В директории `part1-gigachat` расположены:
- `bad_code.py` — исходный пример некачественного кода с глобальным состоянием;
- `refactored_code.py` — улучшенная версия кода;
- `generated_code.py` — функции, сгенерированные моделью по запросу;
- `test_refactored.py` — тесты для валидации работоспособности;
- `README_generated.md` — файл описания, созданный моделью.

### Генерация программного кода

Модель создала три функции, сохраненные в `generated_code.py`:
1. `is_prime(n: int)` — алгоритм проверки простоты числа с оптимизированным перебором делителей.
2. `fibonacci(n: int)` — генерация списка первых `n` чисел Фибоначчи.
3. `normalize_phone(phone: str)` — нормализация телефонных номеров РФ к стандарту `+7XXXXXXXXXX`.

### Рефакторинг и исправление недоработок

В ходе рефакторинга плохой структуры из `bad_code.py` мы избавились от глобальной переменной `data` и неявного поведения. Спроектированы:
- Датакласс `ProductRecord` для хранения информации о товарах (поля `title`, `price_val`, `qty_count`).
- Класс `ShoppingBag`, инкапсулирующий методы добавления продуктов (`add_product`), расчета стоимости (`get_grand_total`) и поиска (`search_item_by_name`).

Это обеспечило слабую связанность (loose coupling) и повысило тестируемость.

### Запуск unit-тестов

Для контроля корректности написаны тесты в `test_refactored.py`, покрывающие основные сценарии использования:
- Расчет итоговой суммы корзины с помощью `ShoppingBag.get_grand_total()`;
- Выброс исключений `ValueError` при передаче некорректных значений цены или количества;
- Проверка сгенерированных GigaChat функций.

### Результаты проверки сетевого подключения

Через вспомогательный скрипт `test_connection.py` было выполнено успешное обращение к API модели с получением краткого статуса работоспособности соединения.

### Оценка качества генерации

Сгенерированные решения содержат аннотации типов и строки документации. Логические ошибки отсутствуют. Однако перед внедрением ИИ-кода в боевое окружение обязательно требуется проведение ручного ревью для исключения скрытых дефектов или уязвимостей.

## Ответы на контрольные вопросы (Часть 1)

1. **Какие типы задач лучше всего решает GigaChat?**  
   Модель показывает отличные результаты при автоматизации написания шаблонных тестов, генерации типовых утилит, первичном рефакторинге и переводе логики с одного языка программирования на другой.

2. **Были ли ошибки в сгенерированном коде?**  
   На этапе проектирования модель могла выдавать неполные проверки граничных условий (например, в нормализации телефонов). Потребовалась коррекция системных промптов для получения точного ответа.

3. **В чем отличие генерации кода через ИИ от использования сниппетов?**  
   Сниппеты статичны и требуют ручной адаптации, в то время как ИИ учитывает контекст задачи и генерирует код с нужными типами данных и именами переменных «на лету».

4. **Какие риски использования ИИ для генерации кода видны на практике?**  
   Главные риски включают генерацию синтаксически верного, но логически неверного кода (галлюцинации), утечку конфиденциальных данных в запросах к сторонним API и отсутствие глубокого понимания архитектуры.

5. **Насколько полезны автоматически сгенерированные тесты?**  
   Они значительно ускоряют написание рутинных тестов на стандартные ситуации (happy path), но сложные тест-кейсы и проверку бизнес-логики разработчик должен проектировать самостоятельно.

## Часть 2. Моделирование CRM-структуры в Airtable

### Схема данных в Airtable

В визуальном интерфейсе Airtable спроектирована схема данных CRM, включающая связанные таблицы:
- `Customers` (клиенты, контакты);
- `Products` (каталог товаров, цены);
- `Orders` (заказы покупателей);
- `Order Items` (строки заказов с привязкой товаров к заказам).

### Экспорт данных через API

Написан скрипт `export_airtable.py`, выгружающий данные по таблицам в CSV-формат. Файлы выгрузки сохранены в папке `exported_data` со специальным суффиксом:
- `customers_sheet.csv`
- `products_sheet.csv`
- `orders_sheet.csv`
- `order_items_sheet.csv`

В выгрузке `customers_sheet.csv` содержится информация о пользователе **Никита Филиппов** из города **Владивосток**.

### Сравнительный анализ подходов

В файле `comparison_table.md` проведено сопоставление Low-Code и традиционного кодинга. Выводы:
- Low-Code идеален для создания MVP, внутренних инструментов и простых баз данных благодаря встроенным формам и интерфейсу;
- Классическая разработка необходима там, где требуются сложные транзакции, высокая производительность под нагрузкой и независимость от проприетарных платформ (отсутствие vendor lock-in).

## Ответы на контрольные вопросы (Часть 2)

1. **Какие ограничения low-code подхода проявились при работе с Airtable?**  
   Это жесткие лимиты на число записей в бесплатном тарифе, ограничения API (скорость запросов и пагинация), а также сложность реализации распределенных транзакций.

2. **Каких типов валидации или возможностей не хватало?**  
   Не хватало сложных условных проверок при вводе данных, ограничений на уровне СУБД (Foreign Key каскадные действия) и гибкого разграничения прав для отдельных полей.

3. **В каких сценариях Airtable эффективнее классической разработки?**  
   При быстром прототипировании, ведении бэклога задач, организации простых опросов, форм регистрации или сбора обратной связи с минимальными затратами времени.

4. **Что такое vendor lock-in и как он проявляется в Airtable?**  
   Это сильная привязка к поставщику услуг. Если Airtable закроется или изменит ценовую политику, миграция бизнес-логики и связей данных в другую систему будет затруднительной.

5. **Как масштабировать такое приложение при росте до очень большого числа заказов?**  
   Необходимо постепенно переносить хранение данных в полноценную СУБД (PostgreSQL / MongoDB), сохраняя Airtable только как один из административных интерфейсов через интеграционные скрипты, либо полностью перейти на классическую backend-архитектуру.
