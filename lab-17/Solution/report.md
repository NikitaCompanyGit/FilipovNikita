## Ответы на контрольные вопросы (Часть 1)

1. **Какие преимущества дает многослойная или продуманная сборка Docker-образа?**  
   Оптимизация Docker-файла позволяет минимизировать размер итогового образа (благодаря multi-stage сборке), изолировать сборочные утилиты от рабочей среды и ускорить сборку за счет кэширования слоев.

2. **Почему frontend размещен на Vercel, а backend в Yandex Cloud?**  
   Vercel предоставляет глобальную сеть доставки контента (CDN), специально оптимизированную под фреймворк Next.js. Yandex Cloud позволяет эффективно хостить контейнеры Docker в виртуальной сети с гибким управлением доступом и базами данных.

3. **Как решается проблема CORS?**  
   На бэкенде настраивается middleware, которое добавляет в HTTP-ответы заголовки `Access-Control-Allow-Origin` со списком разрешенных доменов клиента. Это позволяет веб-браузерам выполнять кросс-доменные запросы к API.

4. **Какие переменные окружения должны быть секретными, а какие могут быть публичными?**  
   Секретными (Secrets) должны оставаться API-ключи, токены авторизации, пароли к БД и приватные ключи сервисных аккаунтов. Публичными могут быть адреса внешних API-эндпоинтов, например `NEXT_PUBLIC_API_URL`, которые компилируются прямо в клиентский JS-код.

5. **Что такое “холодный старт” в serverless-среде?**  
   Это явление, когда при первом обращении к облачной контейнерной функции (после отсутствия нагрузки) требуется время на создание виртуальной машины и запуск контейнера с нуля, что временно увеличивает время ответа.

## Часть 2. Автоматизация процессов CI/CD

### Структура конфигурационных файлов

Для настройки GitHub Actions были написаны и размещены в репозитории три файла сценариев:
1. `filipovLabs/lab-17/Solution/fullstack-app/.github/workflows/deploy.yml` — деплой всего приложения.
2. `filipovLabs/lab-17/Solution/fullstack-app/.github/workflows/pr-checks.yml` — сборка и проверки на Pull Request.
3. `filipovLabs/lab-17/Solution/fullstack-app/.github/workflows/terraform.yml` — проверка IaC-файлов инфраструктуры.

### Описание пайплайна развертывания (`deploy.yml`)

Сценарий запускается автоматически при слиянии изменений с веткой `main`. Он состоит из двух независимых этапов:
- **deploy-frontend**: Выкачивает код, ставит Node.js, устанавливает зависимости, билдит фронт и пушит в Vercel через CLI;
- **deploy-backend**: Логинится в Yandex Container Registry, собирает Docker-образ, пушит его в реестр и обновляет ревизию Serverless-контейнера в облаке.

### Описание пайплайна проверок (`pr-checks.yml`)

Запускается при отправке изменений в ветки Pull Request. Осуществляет предварительную проверку синтаксиса и сборку фронтенда для выявления возможных ошибок слияния на ранних стадиях.

### Описание инфраструктурных проверок (`terraform.yml`)

Интегрирует статический анализ Terraform-конфигураций. Пайплайн запускает `terraform fmt` и `terraform validate` для гарантирования корректности объявленных облачных ресурсов.

## Ответы на контрольные вопросы (Часть 2)

1. **Чем CI/CD пайплайн лучше ручного деплоя?**  
   CI/CD гарантирует повторяемость сборки, исключает ошибки «человеческого фактора», производит автоматические тесты до деплоя и значительно снижает время выкатки обновлений.

2. **Зачем нужны GitHub Actions Secrets?**  
   Для конфиденциального хранения чувствительных ключей доступа и паролей. Они шифруются на серверах GitHub и подставляются в шаги пайплайна в виде замаскированных переменных.

3. **Для чего нужен отдельный workflow на Pull Request?**  
   Чтобы провести автоматическое тестирование нового кода до того, как он будет влит в стабильную ветку `main`. Это предотвращает порчу рабочего кода.

4. **Зачем нужен Terraform validation workflow, если инфраструктура уже создана?**  
   Поскольку инфраструктура описана в виде кода (IaC), любые изменения в terraform-файлах перед развертыванием должны быть проверены линтером на корректность синтаксиса во избежание сбоев в работе облака.

5. **Что происходит, если один из jobs в CI/CD завершается с ошибкой?**  
   Пайплайн останавливается, статус сборки помечается как проваленный, а критические шаги деплоя блокируются, чтобы не допустить выкатку неработающей версии в прод.
