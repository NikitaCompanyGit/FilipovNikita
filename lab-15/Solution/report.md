## Часть 1. ETL-пайплайн на Python

### Исходные данные и структура пайплайна

В первой части лабораторной использовался CSV-файл `lab-15/part1-etl/data/sales.csv`, содержащий данные о продажах интернет-магазина. В файл были включены:

- корректные записи;
- пропуски;
- дубликаты;
- аномальные значения.

Это позволило проверить не только загрузку данных, но и реальную очистку, как в типовом ETL-процессе.

Основной пайплайн реализован в файле `lab-15/part1-etl/etl_pipeline.py`. В нем последовательно выполняются этапы:

1. `extract()`
2. `transform()`
3. `aggregate()`
4. `load_to_sqlite()`
5. `visualize()`

### Выполнение этапа Extract

На этапе извлечения CSV-файл загружается в DataFrame и фиксируется количество строк и колонок.


### Выполнение этапа Transform

На этапе преобразования были выполнены:

- удаление дублей;
- заполнение пропусков;
- преобразование типов;
- фильтрация аномалий;
- расчет новых полей `total_amount` и `month_year`.


Таким образом, из исходных строк:
- была удалена строка-дубль;
- были нормализованы пропуски;
- исключены аномальные значения.

### Агрегация данных

На этапе `aggregate()` выполнялась группировка по категориям и месяцам с расчетом:

- суммарного количества проданных товаров;
- общей выручки;
- средней цены;
- количества уникальных заказов.


### Загрузка в SQLite

Результаты пайплайна были загружены в базу `lab-15/part1-etl/sales.db` в таблицы `sales_cleaned` и `sales_aggregated`.

Проверка содержимого базы подтверждает, что ETL-процесс завершился корректно и привел к созданию реальной витрины данных.


### Визуализация

В процессе выполнения пайплайна были построены графики `revenue_by_category.png`, `monthly_revenue.png`, `category_share.png`. Они сохранены в каталоге `report/graphs/`.

## Ответы на контрольные вопросы (Часть 1)

1. **Какие аномалии и пропуски были обнаружены в исходных данных? Как они были обработаны?**  
   В данных присутствовали дубликаты, пропуски по текстовым и числовым полям, а также аномалии, например некорректные количества (отрицательные) или цены. Дубликаты были удалены, пропуски нормализованы, а строки с аномальными значениями исключены из итоговой выборки.

2. **В чем разница между ETL и ELT?**  
   В ETL данные сначала извлекаются, затем преобразуются и только потом загружаются в хранилище. В ELT данные сначала загружаются как есть, а преобразование выполняется уже внутри целевой системы. Для данной лабораторной был выбран ETL, так как объем данных небольшой и логично очистить их средствами Pandas до загрузки в SQLite.

3. **Почему в реальных проектах данные нельзя загружать в базу без очистки?**  
   Потому что сырые данные содержат ошибки, дубликаты, пропуски и аномалии, которые приводят к искажению аналитики. Без очистки невозможно получить надежные отчеты и корректные бизнес-метрики.

4. **Какие дополнительные трансформации могли бы быть полезны?**  
   Полезными могли бы быть нормализация городов и способов оплаты, проверка выбросов по сумме заказа, категоризация клиентов по среднему чеку, а также построение витрин не только по категориям, но и по городам, клиентам и типам оплаты.

## Часть 2. Apache Kafka

### Подготовка инфраструктуры

Во второй части лабораторной была подготовлена среда Apache Kafka через `docker-compose.yml`. В состав окружения вошли Zookeeper, Kafka и Kafka UI. Это позволило развернуть локальный стенд для потоковой обработки событий заказов.

### Producer: генерация заказов

Файл `lab-15/part2-kafka/producer.py` реализует генерацию заказов интернет-магазина и отправку их в Kafka topic.
Продюсер публикует сообщения с ключом, завязанным на клиента (включая пользователя *Никита Филиппов* из города *Владивосток*), что имитирует реальный поток бизнес-событий.


### Consumer: агрегация статистики

Файл `lab-15/part2-kafka/consumer.py` реализует чтение потока и обновление статистики в реальном времени.

По ходу обработки консьюмер рассчитывает:
- общее количество заказов;
- суммарную выручку;
- распределение заказов по категориям и по городам;
- список последних заказов.


### Windowed consumer

Дополнительно был реализован `windowed_consumer.py`, который строит агрегаты по временному окну (скользящий интервал).


### Практический смысл потоковой части

Потоковая часть моделирует реальный сценарий событийной архитектуры:
- producer публикует события;
- consumer обрабатывает их без ожидания пакетной загрузки;
- windowed consumer строит near-real-time сводки.

Такой подход полезен для систем мониторинга, онлайн-панелей управления и аналитики в реальном времени.

## Ответы на контрольные вопросы (Часть 2)

1. **Что такое topic, partition и offset в Kafka?**  
   Topic - это логический канал сообщений. Partition - отдельная часть topic, позволяющая масштабировать чтение и запись. Offset - порядковый номер сообщения внутри partition, по которому consumer отслеживает, что уже прочитано.

2. **В чем разница между producer и consumer? Зачем нужен `group_id`?**  
   Producer публикует сообщения в Kafka, а consumer читает их и обрабатывает. `group_id` объединяет consumers в одну группу, благодаря чему Kafka распределяет между ними сообщения и обеспечивает масштабируемую обработку без дублирования внутри одной группы.

3. **Какие преимущества потоковая обработка дает по сравнению с ETL?**  
   ETL работает пакетно и обычно обрабатывает уже накопленные данные. Kafka позволяет обрабатывать события сразу после их появления, что делает возможным мониторинг, уведомления и аналитику практически без задержки.

4. **Что произойдет, если запустить два consumer с одинаковым `group_id`?**  
   Kafka воспримет их как одну consumer group и распределит партиции между ними. Это означает, что сообщения будут делиться между экземплярами, а не читаться каждым полностью независимо.

## Сравнение ETL и потоковой обработки

| Характеристика | ETL (Python + SQLite) | Kafka |
|----------------|------------------------|-------|
| Тип обработки | Пакетная | Потоковая |
| Задержка | Минуты/часы | Секунды/миллисекунды |
| Хранение результата | База / витрина | События + агрегаты |
| Удобно для | Отчетности и витрин | Мониторинга и real-time сценариев |
| Сложность инфраструктуры | Ниже | Выше |

Первая часть лабораторной больше подходит для подготовки отчетов и аналитических витрин, а вторая - для обработки событий в живом потоке.
